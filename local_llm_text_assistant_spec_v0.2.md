# Windows常駐型ローカルLLM文章補正ツール 要件定義書 v0.2（開発開始版）

**ステータス:** 開発開始可  
**作成日:** 2026-05-31  
**前バージョンからの主要変更:**
1. マーカー本文挿入を**廃止**（非破壊・読み取りのみに変更）
2. MVPスコープを「一本道」に縮小、常駐/ジョブ管理/履歴をv1.0送り
3. **Phase 0（品質検証）を最優先フェーズとして独立**
4. ターゲットを **RTX 3060 (VRAM 8GB)** に確定、モデル候補を絞り込み
5. **用例コーパス（Corpus）機能**を正式追加 ＝「使ってるうちに進化」の実体
6. クリップボード反映待ち・出力サニタイズをMVP必須に格上げ
7. 受け入れ条件に**意味保存のY/N判定**を追加

---

## 1. 目的

Windows上の任意の入力欄で選択した文章を、ローカルLLMに渡して補正案を生成し、ユーザーが見比べて採否判断できるようにする常駐型支援ツール。

主目的:
- 音声入力・雑なメモを相手に送れる文章へ整える
- タイポ・誤字・脱字・誤変換だけを軽く修正する
- LLM出力を**自動反映せず**、人間が確認して貼り付け・採否判断する
- 外部APIに送らず、ローカルLLMで処理する
- LLM処理中もユーザーが別作業できる

本ツールは自動校正ツールではなく、「選択文を取得し、補正案を生成し、ユーザーが貼り付けて比較する」補助レイヤーである。

---

## 2. 基本コンセプト

### 2.1 操作思想
ユーザーが文章を選択しショートカットを押したときだけ処理する。常時入力監視・入力停止検知・自動変換・自動上書きは初期スコープ外。

### 2.2 安全思想（v0.1から強化）
- LLM生成結果を本文へ自動挿入しない。
- **原文取得は読み取り専用（Ctrl+Cのみ）。本文を書き換えない。** ← v0.1のマーカー挿入を廃止。
- 生成結果はクリップボードに入れ、結果ウィンドウにも表示する。
- ユーザーは元の入力欄で自分で Ctrl+V する。

> **設計判断（v0.1からの転換）:** v0.1はショートカット直後に Ctrl+V を送って選択範囲を `▼▼▼原文▲▲▲` に置換していた。これは「本文を自動で書き換える」操作であり、自動上書き禁止の思想と非対称だった。結果ウィンドウが原文を保持する以上、本文にマーカーを残す必要はない。MVPでは本文を一切触らない。

### 2.3 比較思想
原文と生成文の対比は、**本文ではなく結果ウィンドウ内で**行う。原文ペイン／生成文ペインを並列表示する。

## 3. 対象環境

### 3.1 OS
Windows 10 / Windows 11

### 3.2 ハードウェア（確定）
- **GPU: NVIDIA RTX 3060（VRAM 8GB）を基準とする。**
- モデルは量子化込みで**VRAM 6〜7GBに収まるもの**に限定する。
- CPUフォールバックは可（遅くなる）が、品質検証はGPU前提で行う。

### 3.3 対象アプリ
Ctrl+C / Ctrl+V が標準的に動くアプリ: ブラウザ入力欄、各種チャット入力欄（Slack/Teams/Discord）、メール作成欄、Notepad、VSCode、Obsidian、Markdownエディタ、Word、Google Docs 等。

対象外/制限: 特殊入力欄、管理者権限アプリ、ゲーム、リモートデスクトップ、仮想環境、セキュア入力欄（パスワード欄）。

## 4. 補正モード

### 4.1 初期ショートカット（2つ固定）

| ショートカット | 機能 | 説明 |
|---|---|---|
| Ctrl+Alt+B | ビジネス依頼文化 | 選択文を相手に送れる丁寧な依頼文へ整える |
| Ctrl+Alt+T | タイポ修正 | 誤字・脱字・誤変換・句読点の乱れだけを修正する |

### 4.2 モードA: ビジネス依頼文化
雑な口語を、意味を変えず・事実を追加せず・過剰に堅くせず、相手に送れる依頼文にする。

例: 「この件なんだけど、明日までに確認してもらえると助かります」→「お手数ですが、本件について明日までにご確認いただけますでしょうか。」

### 4.3 モードB: タイポ修正
誤字・脱字・誤変換・明らかな句読点の乱れだけを直す。文体を変えない／言い換えない／丁寧語化しない／要約しない。確信がない場合は無理に推測補正しない。

> **品質上の注意:** 7Bクラスでも「言い換え」「過剰補正」「改行構造の破壊」は起きる。プロンプト制約＋低temperature＋出力サニタイズ＋Phase 0での実測で対処する。「日本語が強いモデルにすれば解決」ではない。

## 5. 操作フロー（MVP・非破壊版）

1. ユーザーが任意アプリで文章を選択する
2. Ctrl+Alt+B または Ctrl+Alt+T を押す
3. 現在のクリップボード内容を退避する
4. Ctrl+C を送信して選択範囲を取得する（**本文は書き換えない**）
5. **クリップボード反映待ち**（ポーリング＋タイムアウト、§7.2）
6. 退避したクリップボードを復元する
7. 取得テキストをジョブとしてローカルLLMへ送る
8. ユーザーは別作業をしてよい
9. LLM完了後、出力をサニタイズする（§8.5）
10. 生成結果をクリップボードに入れる
11. 結果ウィンドウに「原文／生成文」を並列表示する
12. Windows通知を出す
13. ユーザーが元の場所へ戻り、任意位置に Ctrl+V で貼る

本文側には**いかなる痕跡も残さない**。

## 6. 画面要件

### 6.1 結果ウィンドウ（MVP必須）
表示項目: モード名 / 原文ペイン / 生成文ペイン（並列） / コピー状態 / 作成日時。  
ボタン: 「コピー（生成結果を再コピー）」「採用Y/N（§18の品質記録、Phase 1から）」「閉じる」。  
**本文自動挿入ボタンは入れない**（誤爆回避）。

### 6.2 タスクトレイ常駐（Phase 3）
タスクトレイアイコン、有効/無効切替、LLM接続状態、処理中ジョブ数、履歴表示。MVPでは最小（接続状態表示のみ）で可。

### 6.3 完了通知（MVP必須）
例: 「ビジネス依頼文の生成が完了しました。結果をクリップボードにコピーしました。」  
通知クリックで該当ジョブの結果ウィンドウを前面化。

## 7. クリップボード要件

### 7.1 基本動作（非破壊版）
1. 現在のクリップボード内容を退避
2. Ctrl+C 送信で選択範囲取得
3. 反映待ち（§7.2）
4. クリップボードを退避内容へ復元
5. 取得テキストをLLMへ
6. 完了時、生成結果をクリップボードへ（設定で挙動変更可、§7.3）

### 7.2 クリップボード反映待ち（MVP必須・v0.1で欠落）
Ctrl+C 送信直後はクリップボードに前の内容が残っていることがある。
- 送信前のクリップボードハッシュを記録
- ハッシュが変化するまで短間隔でポーリング（例: 20ms間隔）
- タイムアウト（例: 1秒）で「取得失敗」として中止
- 取得値が空 or 非テキストなら中止

### 7.3 上書き挙動（設定可能）
- `copy_result_on_complete: true`（デフォルト）= 完了時に自動でクリップボードへ
- `false` = 結果ウィンドウのコピーボタン押下時のみ（途中コピー保護）
- 上書き発生時は通知で明示し、結果ウィンドウに残し、再コピー可能にする

## 8. LLM要件

### 8.1 ランタイム
ローカルのみ。第一候補 **Ollama**（localhost API）。LM Studio / llama.cpp も `endpoint` 設定で対応可能な構造。外部API送信は禁止。

### 8.2 モデル候補（RTX 3060 / 8GB 確定）

| モデル | 量子化目安 | VRAM目安 | 位置づけ |
|---|---|---|---|
| **Qwen2.5 7B Instruct** | Q4_K_M | ~4.7GB | **第一候補。** 日本語指示追従が4bより明確に強い |
| Gemma 2 9B Instruct | Q4_K_M | ~5.8GB | 日本語の自然さ高。長文選択でVRAMギリギリの懸念 |
| gemma3:4b | — | ~3.3GB | 速度ベースライン。過剰補正リスクあり |
| Qwen2.5 3B Instruct | Q4_K_M | ~2GB | 7Bが遅すぎた時の退避先 |

最終デフォルトは **Phase 0 の実測（品質×速度）で決定**する。設定画面でこの候補群を差し替え可能にする（§16, Phase 2）。

### 8.3 LLMへの入力
原則**選択範囲のみ**を渡す。前後文脈・全文・アプリ名・ウィンドウタイトルは初期スコープでは渡さない（単純化・機密漏えい低減・安定化）。文脈取得は将来の明示オプトイン機能（§16）。

### 8.4 プロンプト要件
- **ビジネス依頼文化:** 意味を変えない／事実追加しない／過剰に堅くしない／長くしすぎない／出力は修正文のみ／解説・前置きを出さない。
- **タイポ修正:** 文体を変えない／言い換えない／丁寧語化しない／要約しない／事実追加しない／**改行・段落構造を維持**／出力は修正文のみ。
- temperature はタイポ修正で低め（例: 0.1〜0.3）をデフォルト。
- プロンプトは設定で編集可能にする（Phase 2）。

### 8.5 出力サニタイズ（MVP必須・v0.1で欠落）
小型モデルは「修正文のみ」と指示しても前置き・装飾を付けがち。クリップボードへ入れる前に除去する:
- 先頭/末尾の前置き（「はい、」「修正しました：」「以下が〜」等）
- 包んでいる引用符 `「」"" '' ` ` ` `
- コードフェンス ``` ```
- 余分な先頭/末尾空白
- ただし**本文内の改行構造は保持**する（タイポ修正で特に重要）

サニタイズ前後の差分はログに残さない（§11.3）。

## 9. Corpus（用例コーパス）要件 ← v0.2新規／「使ってるうちに進化」の実体

> **重要な前提:** ローカルLLMは推論しても重みを学習しない。Ollamaに流すだけでは賢くならない。「使ってるうちに進化」は以下の2方式で実現する。MVP〜v1.0では(A)を採用する。

### 9.1 方式A: 用例コーパス＋Few-shot注入（採用）
ユーザーが採用/編集した「原文→確定文」ペアを蓄積し、次回以降のプロンプトに動的に数例差し込む。重みは変えず、文脈で寄せる。即効・安全・3060で無料。

- 採用フロー: 結果ウィンドウで「採用」を押す、または貼り付け後に編集した最終文を任意で登録する
- コーパス項目: `{ mode, source_text, accepted_text, created_at, tags }`
- Few-shot選択: モード一致＋（将来）類似度上位N件をプロンプトへ注入
- 件数上限・手動編集・削除をユーザーが管理できる
- **全文貼り付けによる一括投入**: ユーザーが過去の自分の文章群を貼り付け、モード別に用例として取り込めるインポート機能（Phase 2）

### 9.2 方式B: LoRAファインチューニング（将来オプション・別プロジェクト級）
コーパスが十分量たまった段階で、QLoRAにより7Bを実際に追加学習する。3060 8GBでもQLoRAは技術的に可能だが、データ整形・学習ループ・評価・破綻検知が必要。MVPには含めない。

> **釘:** ファインチューニングは「文体を覚える」一方で「タイポ修正で勝手に言い換える癖」も覚えうる。導入時は保持/言い換えの評価を必須にする。安易に方式Aから方式Bへ飛ばない。

### 9.3 プライバシー
コーパスは**ローカル保存のみ・外部送信なし**。保存ON/OFF、保存先、暗号化、プロジェクト別分割をユーザーが選べる（既定はローカル平文・ユーザー任意でON）。

## 10. エラー処理

| ケース | 動作 |
|---|---|
| 選択範囲なし | 中止＋通知「文章を選択してから再実行してください」 |
| クリップボード反映タイムアウト | 中止＋通知（§7.2） |
| LLM未接続 | 中止＋通知「ローカルLLMに接続できませんでした。Ollama/設定を確認してください」 |
| 生成失敗 | 結果ウィンドウに失敗内容表示＋通知 |
| 文字数超過 | 初期は拒否（既定上限3000字）＋通知。将来チャンク分割 |
| 非テキスト取得 | 中止 |

v0.1にあった「貼り付け(Ctrl+V)失敗」「マーカー挿入失敗」は、非破壊化により**廃止**。

## 11. 非機能要件

### 11.1 応答性
- ショートカット押下→取得完了: 1秒以内目標（反映待ちタイムアウト含む）
- 生成時間（3060/7B目安・Phase 0で実測更新）: タイポ修正 3〜10秒、依頼文化 5〜20秒
- 生成中もユーザーの別作業を妨げない

### 11.2 安全性
常時キーログ保存なし／入力全体監視なし／選択文のみ処理／外部API送信なし／テキスト限定／セキュア入力欄は対象外。

### 11.3 プライバシー
初期既定: 結果履歴はメモリのみ、終了時破棄、本文をログに保存しない。エラーログはジョブID・モード・時刻・エラー種別のみ。コーパスは§9.3に従う。

### 11.4 可用性
LLM停止中でもツールは起動可。未接続状態を表示。

### 11.5 拡張性
ショートカット/モード/モデル/プロンプト/文体プリセット/アプリ別設定/文脈取得/差分表示/拡張連携を後から追加できる構造。

## 13. 技術アーキテクチャ

```
[User] → [Global Hotkey] → [Selection Capture (Ctrl+C, 非破壊)]
       → [Clipboard Wait + Restore] → [Job Queue]
       → [Prompt Router (+Corpus few-shot)] → [Local LLM Client (Ollama)]
       → [Output Sanitizer] → [Result Handler (clipboard + window + notify)]
       → [User: Ctrl+V で任意位置へ貼り、比較・採否]
```

| コンポーネント | 役割 |
|---|---|
| Hotkey Manager | ショートカット検知 |
| Clipboard Manager | 退避・取得・反映待ち・復元・結果コピー |
| Job Queue | ジョブ管理（MVPは逐次1件で可） |
| Prompt Router | モード別プロンプト選択＋Corpus注入 |
| Local LLM Client | Ollama API 呼び出し |
| Output Sanitizer | 前置き・引用符・フェンス除去、改行保持 |
| Corpus Store | 用例の保存・検索・インポート |
| Result Window | 原文／生成文 並列表示、採否記録 |
| Notifier | Windows通知 |
| Settings | 設定の保持・編集 |

## 14. 技術選定

- **Phase 0〜3 プロトタイプ:** AutoHotkey v2 ＋ ローカルPythonワーカー ＋ Ollama。  
  → ホットキー検知・送信は AHK、LLM呼び出し・サニタイズ・Corpus・結果ウィンドウは Python。
- **Phase 4 本実装（育てる判断時）:** C# / .NET ＋ WPF or WinUI ＋ Ollama。  
  → **移植＝実質作り直し**と割り切る。AHKのロジックはC#へ流用できない前提。プロトタイプは「挙動を固めて捨てる」もの。

## 15. 開発ロードマップ

### Phase 0 — 品質検証（最優先・コードのガワを作らない）
目的: **補正品質が実用に足るか**を、本体実装前に潰す。
- Ollama に Qwen2.5 7B を入れ 3060 で起動
- §8.4 のタイポ修正/依頼文化プロンプトを、実文章20〜30件に手で流す
- 「意味保存できているか」をY/Nで記録、言い換え・過剰補正・改行破壊の発生率を測る
- 7B / 9B / 4b / 3B を並べ、品質×速度を比較
- 成果物: 確定プロンプト、暫定デフォルトモデル、生成速度実測値、品質基準
- **ここで品質が死んでいたらアプリを作らない。**

### Phase 1 — 一本道MVP（非破壊）
- グローバルホットキー2つ
- 選択文取得（Ctrl+Cのみ／反映待ち＋タイムアウト）
- Ollama呼び出し
- **出力サニタイズ**
- 結果をクリップボードへ＋結果ウィンドウに原文/生成文並列表示
- 完了/エラー通知、本文ログ保存なし、採否Y/N記録
- 成果物: 自分で毎日使える最小ツール

### Phase 2 — 設定画面 + モデル切替 + Corpus
- 設定画面（ホットキー/エンドポイント/モデル選択ドロップダウン/temperature/プロンプト編集）
- §8.2候補からのモデル入れ替え、モデル別の速度・VRAM表示
- **Corpus（方式A）**: 採用ペア蓄積、few-shot注入、**全文貼り付けによる一括インポート**
- 成果物: 「軽くて日本語が強いものに差し替え可能」＋「使うほど寄っていく」を満たす

### Phase 3 — 常駐アプリ化と堅牢性
- タスクトレイ常駐、有効/無効、処理中ジョブ数、履歴（メモリのみ）
- ジョブID体系、並列実行制限（1〜2）
- 対象外アプリのガード、クリップボード退避・復元の安定化
- 成果物: v1.0（人に渡せる常駐ツール）

### Phase 4 — C#/.NET移植（育てる判断をしたら）
- C#/.NET + WPF/WinUI + Ollama、UI Automationで取りこぼし改善
- 永続履歴（暗号化）、文体プリセット土台

### Phase 5 — 拡張
モード追加、文脈取得（明示オプト）、差分表示、置換挿入、Office/ブラウザ拡張、**Corpus方式B（LoRA）検討**。

## 16. MVP範囲（Phase 1）

**必須:** ホットキー2つ／選択文取得（反映待ち）／Ollama呼び出し／出力サニタイズ／クリップボードコピー／結果ウィンドウ並列表示／完了・エラー通知／本文ログ非保存／採否Y/N記録。

**やらない:** 自動上書き／完了時自動貼り付け／本文マーカー挿入／入力停止検知／常時監視／IME統合／アプリ別最適化／文脈取得／全文要約／差分表示／クラウドLLM／履歴永続化／Corpus（Phase 2）／設定画面（Phase 2）／Office・ブラウザ・VSCode拡張。

## 18. 受け入れ条件

### 18.1 ビジネス依頼文化
選択文を取得できる／本文を書き換えない／Ollamaに送れる／**出力がサニタイズ済みで前置き・引用符を含まない**／依頼文として自然／完了通知が出る／クリップボードに入る／結果ウィンドウに原文と並列表示／本文へ自動挿入しない。

### 18.2 タイポ修正
選択文を取得できる／本文を書き換えない／タイポ修正プロンプトで処理／**文体が大きく変わらない／改行構造が保持される**／サニタイズ済み／完了通知／クリップボード／結果ウィンドウ表示。

### 18.3 意味保存判定（v0.2新規）
結果ウィンドウで、ユーザーが「原文と意味が一致しているか」を **Y/N で記録**できる。Phase 0/1の品質評価はこのY/N率を指標とする。

### 18.4 安全性
常時キーログ保存なし／選択文以外を取得しない／**本文を書き換えない**／外部API送信なし／本文を永続ログに保存しない／完了時に自動貼り付けしない。
